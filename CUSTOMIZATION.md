# Настройка и расширение пайплайна

Это руководство для тех, кто хочет адаптировать систему проверки под свои правила и условия: добавить запрещённые слова, изменить стайлгайд, подключить другую модель или перестроить систему под другую структуру документации.

---

## Правила линтера

Линтер читает правила из YAML-файлов в папке `styles/Russian/`. Сейчас там два файла:

- `WordChoice.yml` — запрещённые слова: размытые формулировки, канцеляризмы, маркетинговые прилагательные, антропоморфизмы.
- `Substitutions.yml` — директивные и модальные конструкции: «вы должны», «запрещено», «необходимо».

### Структура YAML-файла

```yaml
extends: existence          # тип правила — не менять
message: "Текст сообщения: '%s'. Подсказка."
level: warning              # severity: warning или error
nonword: true               # не требовать границы слова — нужно для кириллицы

raw:                        # список паттернов (регулярные выражения Python)
  - 'паттерн 1'
  - 'паттерн 2'

raw_hints:                  # подсказки замены — параллельный список к raw
  - 'замена для паттерна 1'
  - 'замена для паттерна 2'
```

Каждый элемент `raw_hints[i]` соответствует `raw[i]`. Если подсказки нет — оставьте пустую строку `''`.

### Как добавить запрещённое слово

Откройте `styles/Russian/WordChoice.yml`. В нужном разделе добавьте паттерн в список `raw` и подсказку в список `raw_hints` на той же позиции.

Пример — добавить слово «интуитивный» с подсказкой:

```yaml
raw:
  # ... существующие паттерны ...
  - '[Ии]нтуитивн\w*'       # ← новая строка

raw_hints:
  # ... существующие подсказки ...
  - 'опишите конкретно, что именно просто в использовании'   # ← новая строка
```

Обратите внимание на `\w*` в конце — это позволяет поймать все падежные формы: «интуитивный», «интуитивного», «интуитивным», «интуитивными».

### Паттерны для кириллицы

Несколько правил, которые упрощают написание паттернов:

**Регистр.** Указывайте обе формы в квадратных скобках в начале: `[Ии]нтуитивн\w*` поймает и «Интуитивный» в начале предложения, и «интуитивный» в середине.

**Падежные формы прилагательных.** Окончание `\w*` покрывает все формы. Не нужно перечислять `(ый|ого|ому|ым|ом|ая|ой|ую|ое|ые|ых|ым|ыми)` вручную.

**Пробелы и предлоги.** Если нужно поймать фразу с предлогом — пишите пробел явно: `'[Вв] большинстве случаев'`. Пробел в регулярном выражении работает буквально.

**Несколько слов одним паттерном.** Можно объединить через `|`: `'[Мм]ожно (сказать|отметить|добавить)'`.

### Как протестировать новое правило локально

Запустите линтер вручную из корня репозитория:

```bash
python .github/scripts/ru_linter.py \
  --files docs/ru/ваш-файл.mdx \
  --output /tmp/result.json \
  --styles-dir ./styles/Russian
cat /tmp/result.json
```

Если правило сработало — в JSON увидите запись с номером строки и сообщением. Если нет — проверьте паттерн на [regex101.com](https://regex101.com), выбрав Python в качестве движка.

### Создать новый файл правил

Если хотите разделить правила по категориям — создайте новый YAML-файл в `styles/Russian/`. Линтер автоматически загрузит все `.yml`-файлы из этой папки.

Скопируйте структуру из существующего файла и заполните своими паттернами. Имя файла станет частью имени правила: файл `Tone.yml` создаст правило `Russian.Tone`.

## Стайлгайд для LLM

Стайлгайд живёт в папке `style_guide/` — пять markdown-файлов. LLM получает их содержимое целиком при каждой проверке и использует как инструкцию для анализа.

### Как редактировать существующий раздел

Откройте нужный файл (`01_instructions.md`, `02_neutral_tone.md` и т.д.) и отредактируйте текст. Пишите правила конкретно — не «избегайте пассивного залога», а «используйте активный залог: "нажмите кнопку", а не "кнопка должна быть нажата"».

LLM лучше работает с конкретными примерами. Хорошая структура для правила:

```markdown
### Название правила

Что нарушает правило и почему это проблема.

**Нельзя:** пример неправильного текста.  
**Нужно:** пример правильного текста.
```

### Как добавить новый раздел стайлгайда

1. Создайте файл с порядковым номером: `style_guide/06_новый_раздел.md`.
2. Напишите правила в том же формате, что и существующие файлы.

Линтер подхватит новый файл автоматически — он загружает все файлы по маске `0*.md` из папки `style_guide/`.

### Ограничения по объёму

LLM получает не более 50 000 символов стайлгайда. Сейчас весь стайлгайд занимает около 34 000 символов — запас есть. Если стайлгайд вырастет, константу можно изменить в `.github/scripts/review-agent.py`:

```python
MAX_GUIDE = 50_000   # строка ~270 в review-agent.py
```

## Модель LLM

### Сменить модель

Измените переменную `LLM_MODEL` в GitHub → Settings → Secrets and variables → Actions → Variables.

Примеры значений:

| Провайдер | Значение LLM_MODEL | Примечание |
|---|---|---|
| OpenAI | `gpt-4o-mini` | Быстро, дёшево, достаточно для большинства задач |
| OpenAI | `gpt-4o` | Лучше для сложных текстов, дороже |
| Anthropic | `claude-3-5-haiku-20241022` | Через OpenAI-совместимый прокси |
| Корп. LLM | `openai/gpt-oss-120b` | Зависит от вашего провайдера |

### Подключить другого провайдера

Агент использует OpenAI-совместимый API. Любой провайдер с совместимым интерфейсом (Anthropic через прокси, Azure OpenAI, Mistral и др.) подключается через переменную `LLM_BASE_URL`.

В GitHub Variables задайте:
- `LLM_BASE_URL` — базовый URL эндпоинта провайдера
- `LLM_MODEL` — имя модели в формате вашего провайдера
- `LLM_API_KEY` (Secret) — ключ доступа

Если `LLM_BASE_URL` не задана — агент использует `api.openai.com`.

### Изменить параметры проверки

В файле `.github/scripts/review-agent.py` есть константы в начале файла:

```python
MAX_INLINE     = 20       # максимальное число inline-комментариев на PR
MAX_FILES      = 10       # максимальное число файлов на один прогон
MAX_DIFF_CHARS = 60_000   # максимальный размер diff (символы)
MAX_FILE_CHARS = 30_000   # максимальный размер одного файла для LLM
MAX_GUIDE      = 50_000   # максимальный размер стайлгайда
```

## Структура папок с документацией

По умолчанию воркфлоу проверяет файлы по шаблону `docs/(ru|en)/*.mdx`. Если ваша документация лежит в другом месте, измените шаблон в `.github/workflows/docs-review.yml`:

```yaml
CHANGED=$(git diff --name-only "origin/${BASE}...HEAD" \
  | grep -E '^docs/(ru|en)/.+\.mdx$' \   # ← изменить этот паттерн
  || true)
```

Примеры альтернативных паттернов:

```bash
# Только один язык
'^content/ru/.+\.md$'

# Несколько форматов
'^docs/(ru|en)/.+\.(md|mdx)$'

# Без языкового подраздела
'^docs/.+\.mdx$'
```

Паттерн — регулярное выражение, которое передаётся в `grep -E`.

### Если русские файлы лежат не в docs/ru/

В том же файле найдите шаг линтера и измените фильтр:

```yaml
RU_FILES=$(grep 'docs/ru' "$RUNNER_TEMP/changed_files.txt" ...)  # ← изменить путь
```

И аналогично для английских файлов:

```yaml
EN_FILES=$(grep 'docs/en' "$RUNNER_TEMP/changed_files.txt" ...)  # ← изменить путь
```

## Добавить язык

Если нужно проверять документацию на третьем языке:

1. Создайте папку для правил, например `styles/German/`.
2. Добавьте YAML-файлы с правилами по аналогии с `styles/Russian/`.
3. В воркфлоу добавьте новый шаг линтера по образцу шага 5а, указав путь к новым файлам и папке назначения.
4. Добавьте новую папку в шаг объединения результатов (шаг 5в).

Для языков с латиницей проще использовать Vale: создайте `.vale-de.ini` по аналогии с `.vale-en.ini` и добавьте шаг проверки по образцу шага 5б.
