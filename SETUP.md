# Настройка автоматической проверки документации

Это руководство поможет настроить пайплайн проверки документации после форка репозитория.

Пайплайн состоит из трёх компонентов:
- **Python-линтер** — проверяет русские файлы по словарным правилам
- **Vale** — проверяет английские файлы
- **LLM-агент** — читает изменённые файлы целиком и оставляет комментарии в PR

После настройки процесс для технического писателя выглядит так:  
открыть PR → повесить лейбл `docs-review` → получить комментарии с замечаниями.

---

## Что нужно настроить

| Компонент | Обязательно | Примечание |
|---|---|---|
| GitHub Actions runner | Да | Self-hosted (вариант A) или GitHub-hosted (вариант B) |
| LLM API ключ | Да | Корпоративный или публичный (OpenAI, etc.) |
| MCP-сервер | Нет | Без него система работает через локальные файлы |

Выберите вариант в зависимости от ваших условий.

---

## Вариант A: Self-hosted runner

**Когда использовать:** работаете с конфиденциальными данными, хотите использовать LLM внутри корпоративного периметра или уже используете корпоративный LLM-эндпоинт.

### Требования к машине

- Windows 10/11 или Windows Server 2019+
- Python 3.8 или выше (`python --version`)
- Git (`git --version`)
- Постоянный доступ в интернет (для GitHub API, MCP-сервера)
- Если используете корпоративный LLM — доступ к корп. сети

### Шаг 1. Установить и зарегистрировать раннер

1. Откройте ваш форкнутый репозиторий на GitHub.
2. Перейдите в **Settings → Actions → Runners → New self-hosted runner**.
3. Выберите **Windows**.
4. Следуйте инструкциям на экране: скачайте архив, распакуйте в `C:\actions-runner`, выполните команды настройки в PowerShell.

При настройке раннер спросит имя и группу — можно оставить значения по умолчанию.

### Шаг 2. Установить раннер как службу Windows

Запустить раннер вручную через `run.cmd` — ненадёжно: он останавливается при закрытии терминала или перезагрузке машины. Правильный способ — установить как службу.

В PowerShell от имени администратора, из папки `C:\actions-runner`:

```powershell
.\svc.sh install
.\svc.sh start
```

После этого раннер запускается автоматически при старте Windows и не требует ручного вмешательства.

Проверить статус:

```powershell
.\svc.sh status
```

Должно быть `Active: running`.

### Шаг 3. Настроить LLM

#### Вариант A1: корпоративный LLM

Запросите у администратора вашего LLM-сервиса:
- URL эндпоинта API (формат: `https://your-llm.company.ru/v1`)
- API-ключ

#### Вариант A2: публичный OpenAI

1. Зарегистрируйтесь на [platform.openai.com](https://platform.openai.com).
2. Перейдите в **API Keys → Create new secret key**.
3. Скопируйте ключ — он показывается только один раз.

### Шаг 4. Добавить секреты и переменные в GitHub

Откройте ваш форкнутый репозиторий → **Settings → Secrets and variables → Actions**.

**Secrets** (нажмите «New repository secret» для каждого):

| Имя | Значение |
|---|---|
| `LLM_API_KEY` | API-ключ вашего LLM |
| `MCP_API_KEY` | Токен MCP-сервера (если настраиваете MCP; иначе пропустите) |

**Variables** (вкладка Variables → «New repository variable»):

| Имя | Значение | Обязательно |
|---|---|---|
| `LLM_MODEL` | Имя модели, например `gpt-4o-mini` или `openai/gpt-4o` | Да |
| `LLM_BASE_URL` | URL эндпоинта, например `https://your-llm.company.ru/v1` | Только для корп. LLM; для OpenAI не нужна |
| `MCP_SERVER_URL` | URL вашего MCP-сервера | Нет |

### Шаг 5. Создать лейбл docs-review

В репозитории → **Issues → Labels → New label**.  
Имя: `docs-review`. Цвет — любой.

Без этого лейбла воркфлоу не запустится.

## Вариант B: GitHub-hosted runner и публичный API

**Когда использовать:** нет доступа к корпоративному LLM, хотите минимальную инфраструктуру, не работаете с конфиденциальными данными.

GitHub-hosted runner — это облачная виртуальная машина, которую GitHub запускает автоматически. Вам не нужно ничего устанавливать на своём компьютере.

> **Примечание по стоимости.** Для публичных репозиториев GitHub Actions бесплатны. Для приватных — бесплатно 2 000 минут в месяц на Free-тарифе. Запуск одной проверки занимает 3–5 минут.

### Шаг 1. Получить API-ключ OpenAI

1. Зарегистрируйтесь на [platform.openai.com](https://platform.openai.com).
2. Перейдите в **API Keys → Create new secret key**.
3. Пополните баланс — минимальная сумма $5. Одна проверка стоит около $0.01–0.05 в зависимости от модели и объёма файла.
4. Скопируйте ключ.

### Шаг 2. Изменить конфигурацию воркфлоу

Откройте файл `.github/workflows/docs-review.yml` и внесите два изменения.

**Изменение 1 — runner.** Найдите строку:

```yaml
    runs-on: self-hosted
```

Замените на:

```yaml
    runs-on: ubuntu-latest
```

**Изменение 2 — Vale binary.** Найдите блок установки Vale:

```yaml
          curl -sL --max-time 120 \
            "https://github.com/errata-ai/vale/releases/download/v${VALE_VERSION}/vale_${VALE_VERSION}_Windows_64-bit.zip" \
            -o "$RUNNER_TEMP/vale.zip"
          unzip -o "$RUNNER_TEMP/vale.zip" -d "$RUNNER_TEMP/vale-bin"
```

Замените на:

```yaml
          mkdir -p "$RUNNER_TEMP/vale-bin"
          curl -sL --max-time 120 \
            "https://github.com/errata-ai/vale/releases/download/v${VALE_VERSION}/vale_${VALE_VERSION}_Linux_64-bit.tar.gz" \
            -o "$RUNNER_TEMP/vale.tar.gz"
          tar -xzf "$RUNNER_TEMP/vale.tar.gz" -C "$RUNNER_TEMP/vale-bin"
```

**Изменение 3 — python → python3.** В том же файле замените все вхождения `python .github/` и `python -` на `python3 .github/` и `python3 -` соответственно.

Сохраните и запушьте изменения в `main`.

### Шаг 3. Добавить секреты и переменные в GitHub

Откройте ваш форкнутый репозиторий → **Settings → Secrets and variables → Actions**.

**Secrets:**

| Имя | Значение |
|---|---|
| `LLM_API_KEY` | Ваш ключ OpenAI |

**Variables:**

| Имя | Значение |
|---|---|
| `LLM_MODEL` | `gpt-4o-mini` (баланс цена/качество) или `gpt-4o` |

Переменную `LLM_BASE_URL` не добавляйте — агент автоматически возьмёт эндпоинт `api.openai.com`.

### Шаг 4. Создать лейбл docs-review

В репозитории → **Issues → Labels → New label**.  
Имя: `docs-review`. Цвет — любой.

## Опционально: MCP-сервер

MCP-сервер централизованно хранит стайлгайд и правила линтера. Без него система работает через локальные файлы в репозитории — для тестирования этого достаточно.

Если хотите настроить MCP-сервер:

1. Зарегистрируйтесь на [render.com](https://render.com).
2. Нажмите **New → Web Service**, выберите ваш репозиторий. Render прочитает `render.yaml` автоматически.
3. В настройках сервиса добавьте переменную окружения `MCP_API_KEY` — придумайте произвольный токен.
4. Нажмите **Deploy**.
5. После деплоя скопируйте URL сервиса (формат `https://ваш-сервис.onrender.com`).
6. Добавьте в GitHub:
   - Secret `MCP_API_KEY` — тот же токен, что задали на Render
   - Variable `MCP_SERVER_URL` — URL сервиса

## Проверка

1. Создайте ветку от `main`, внесите любое изменение в файл в папке `docs/ru/` или `docs/en/`.
2. Откройте Pull Request.
3. Повесьте на PR лейбл `docs-review`.
4. Перейдите во вкладку **Actions** — должен появиться запущенный воркфлоу **Docs Review**.
5. После завершения (2–5 минут) в PR появятся комментарии.

### Ожидаемые сообщения в логах

```
[INFO] Loaded N rule(s) from MCP: [...]      # если MCP настроен
[INFO] Using local rules from disk.           # если MCP не настроен — нормально
[OK] Style guide loaded from MCP server       # если MCP настроен
[INFO] MCP unavailable, using local style guide  # если MCP не настроен — нормально
[OK] Review posted with N inline comment(s).
```

## Если что-то пошло не так

**Воркфлоу не запускается после лейбла**  
Убедитесь, что лейбл называется именно `docs-review` (без пробелов, строчными буквами).

**Вариант A: воркфлоу зависает в очереди**  
Раннер не запущен. На машине с раннером откройте PowerShell и выполните:
```powershell
cd C:\actions-runner
.\svc.sh status
.\svc.sh start   # если статус не running
```

**LLM возвращает Connection error**  
- Вариант A с корп. LLM: проверьте, что машина с раннером имеет доступ к корп. сети.
- Вариант B: проверьте, что `LLM_API_KEY` добавлен в Secrets и переменная `LLM_BASE_URL` не задана (или удалите её).

**Линтер загружает 0 правил**  
Убедитесь, что в папке `styles/Russian/` есть файлы `WordChoice.yml` и `Substitutions.yml`. Если настроен MCP — проверьте доступность сервера через `/health`.

**GitHub API 422 при публикации комментариев**  
Воркфлоу автоматически переключится на публикацию только сводки без inline-комментариев. Это нормальный fallback.
