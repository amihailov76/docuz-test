Масштабирование и воспроизводимость

Анализ двух треков с ограничениями и планом реализации

Зависимость между треками

Оба трека упираются в одну и ту же проблему: self-hosted раннер на Windows. Он непредсказуемо падает, привязан к конкретной машине и делает воркфлоу нерабочим для коллег-форкеров и неудобным при масштабировании на несколько репозиториев. Переход на GitHub-hosted runner (ubuntu-latest) — общее решение для обоих треков. Его стоит сделать первым, до начала любого другого трека.

A Масштабирование: reusable workflows
Суть подхода

Создаётся служебный репозиторий (docs-automation), который содержит весь механизм проверки: воркфлоу, Python-скрипты, Vale-конфиги. Продуктовые репозитории хранят только минимальный «вызывающий» воркфлоу — 10–15 строк. Всё остальное берётся из служебного репо.

Правила линтера и стайлгайд уже живут на MCP-сервере и доступны всем репозиториям без каких-либо изменений.

Сценарий работы
Этап 1 — триггер
Писатель вешает лейбл docs-review на PR в продуктовом репо

Короткий caller-воркфлоу вызывает reusable workflow из docs-automation. Секреты (LLM_API_KEY, MCP_API_KEY) передаются через org-level secrets — настраиваются один раз для всей организации.

Этап 2 — подготовка окружения
Reusable workflow клонирует оба репозитория

Сначала — продуктовый репо (для docs-файлов и диффа PR), затем — docs-automation в подпапку (для Python-скриптов и Vale-конфигов). Vale скачивается под Linux.

Этап 3 — проверка
Линтер и LLM-агент работают штатно

Линтер берёт правила с MCP-сервера. LLM-агент берёт стайлгайд с MCP-сервера. Скрипты — из подпапки docs-automation. Результат — комментарии в PR продуктового репо.

Ограничения и риски
Blocker self-hosted runner на Windows. Reusable workflow наследует раннер из служебного репо. Если там self-hosted — все продуктовые репо зависят от одной машины. Решение: перейти на ubuntu-latest перед реализацией трека.
Риск Доступность служебного репозитория. Если docs-automation приватный, каждый продуктовый репо должен иметь право клонировать его. Для публичного — проблемы нет.
Важно Структура docs/ должна совпадать. Текущий воркфлоу ищет файлы по шаблону docs/(ru|en)/*.mdx. Если в продуктовом репо структура другая — нужен input-параметр в reusable workflow.

B Воспроизводимость: репо для форка
Целевой сценарий

Коллега-писатель находит репозиторий, делает форк и в течение разумного времени получает работающую систему проверки документации — без звонков тебе, без настройки твоей машины, без доступа к твоим аккаунтам.

Узкие места — анализ
Blocker Self-hosted runner на твоей машине. Форкнувший пользователь не имеет доступа к твоему раннеру — воркфлоу зависнет в очереди навсегда. Единственное решение: ubuntu-latest. GitHub-hosted runners бесплатны для публичных репо и тратят minutes для приватных (2000 минут/месяц бесплатно на Free-тарифе).
Риск MCP-сервер зарегистрирован на твою учётку Render. Форкнувший не сможет использовать твой сервер (и не должен: это твои правила, твой аккаунт). Но система уже умеет работать без MCP — через локальный fallback. Решение: документировать два режима. Базовый (без MCP, всё в репо) — работает из коробки. Полный (с MCP) — требует деплоя своего сервера.
Риск LLM API ключ. Это неизбежно: у каждого пользователя должен быть свой ключ OpenAI или другого провайдера. Решение: чёткая документация, ссылки, объяснение что делать если ключа нет (system работает без LLM — только линтер).
Важно Тестовые файлы в корне репо. test_cyrillic.md, test_vale.md, test_vale.mdx — артефакты разработки, вводят в заблуждение. Убрать или добавить в .gitignore.
Уже OK Fallback везде. Линтер, LLM-агент, MCP — все компоненты деградируют gracefully. Если чего-то нет — работает без него. Это хорошая база для воспроизводимости.
Два режима после форка
Режим Что нужно настроить Что работает
Базовый
«Просто проверить что работает»		 Только LLM_API_KEY в GitHub Secrets Линтер по локальным правилам + LLM по локальному стайлгайду
Полный
«Хочу MCP-сервер»		 LLM_API_KEY + деплой своего MCP на Render + MCP_SERVER_URL, MCP_API_KEY Всё: линтер и LLM берут данные с центрального сервера
Структура документации

Документация должна покрывать два уровня: «что делать» (для писателей) и «как устроено» (для тех, кто хочет настроить или расширить).

ФайлАудиторияСодержание
README.md Все Обзор: что делает репозиторий, как запустить проверку (лейбл на PR), ссылки на остальные доки
docs/SETUP.md Кто форкает Пошаговая настройка: форк → секрет → тест. Базовый и полный режим.
docs/HOW-IT-WORKS.md Любопытные писатели Архитектура пайплайна: линтер, Vale, LLM, MCP. Что происходит после лейбла.
docs/CUSTOMIZATION.md Продвинутые пользователи Как редактировать правила линтера, стайлгайд, модель LLM, структуру папок
