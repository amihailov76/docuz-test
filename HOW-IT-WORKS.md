# Как работает автоматическая проверка документации

Этот документ объясняет устройство пайплайна — что происходит после того, как автор вешает лейбл `docs-review` на Pull Request, и из каких компонентов состоит система.

---

Технический писатель, как правило, получает обратную связь по качеству текста только на этапе ревью — от редактора или коллеги. Это поздно: правки требуют дополнительного раунда согласования, а часть замечаний носит механический характер и не требует экспертизы человека.

Пайплайн решает эту проблему: он проверяет документацию автоматически в момент создания Pull Request. Линтер находит нарушения словарных правил — запрещённые слова, канцеляризмы, директивные конструкции. LLM-агент проверяет то, что линтер не может: структуру инструкций, нейтральность тона, ясность формулировок. Автор получает конкретные замечания со ссылкой на строку и объяснением, что именно нарушено и как исправить, — до того как текст уходит на ревью к человеку.

## Общая схема

```
Автор вешает лейбл docs-review на PR
            │
            ▼
  GitHub Actions запускает воркфлоу
            │
            ├─► Определяем изменённые файлы
            │
            ├─► Линтер (Python) ──── русские .mdx
            ├─► Vale ─────────────── английские .mdx
            │
            ├─► LLM-агент
            │     ├── читает полный текст изменённых файлов
            │     ├── получает стайлгайд
            │     └── получает результаты линтера
            │
            └─► Публикует комментарии в PR
```

---

## Компоненты

### GitHub Actions воркфлоу

**Файл:** `.github/workflows/docs-review.yml`

Воркфлоу — это оркестратор всего процесса. Он описывает последовательность шагов и условия их выполнения. Запускается двумя способами:

- Автоматически: когда на PR вешают лейбл `docs-review`
- Вручную: через вкладку Actions → Run workflow (удобно для отладки)

Воркфлоу проверяет только файлы, которые изменились в PR, — не всю документацию целиком. Это ускоряет работу и снижает количество шума.

---

### Python-линтер

**Файл:** `.github/scripts/ru_linter.py`  
**Что проверяет:** русские `.mdx`-файлы в `docs/ru/`

Линтер читает правила из YAML-файлов и ищет запрещённые паттерны в тексте. Он пропускает фронтматтер, блоки кода и JSX-компоненты, чтобы не генерировать ложные срабатывания.

Для каждого найденного нарушения линтер указывает:
- номер строки
- какое правило нарушено
- что именно написано
- чем рекомендуется заменить

Результат линтера сохраняется в JSON-файл и передаётся LLM-агенту — агент видит, что уже нашёл линтер, и не дублирует те же замечания.

**Почему не Vale для русского?** Vale использует движок Go RE2, который не поддерживает Юникод на Windows так же, как Python. Паттерны с кириллицей на Windows-раннере не срабатывают. Поэтому для русских файлов написан отдельный линтер на Python, который читает те же YAML-правила, но обрабатывает их через Python `re` с поддержкой Юникода.

---

### Vale

**Файлы:** `.vale-en.ini`, `styles/`  
**Что проверяет:** английские `.mdx`-файлы в `docs/en/`

Vale — стандартный инструмент для проверки технических текстов. Он применяет правила из пакетов Microsoft и proselint: пассивный залог, сложные слова с более простыми синонимами, стилистические паттерны.

Пакеты Vale закоммичены прямо в репозиторий (`styles/`), поэтому не требуют доступа к интернету при запуске.

---

### Правила линтера

**Папка:** `styles/Russian/`

Два файла с правилами:

- **`WordChoice.yml`** — запрещённые слова и выражения: размытые формулировки («как правило», «в большинстве случаев»), маркетинговые прилагательные («мощный», «гибкий», «уникальный»), канцеляризмы («осуществить», «произвести»), антропоморфизмы («система думает», «программа понимает»).

- **`Substitutions.yml`** — директивные и модальные конструкции: «вы должны» → «вам нужно», «запрещено» → «не поддерживается», «невозможно» → «недоступно».

Каждый паттерн содержит подсказку с правильной заменой — она показывается в комментарии к PR.

**Как добавить новое правило:** откройте нужный YAML-файл, добавьте паттерн в список `raw` и соответствующую подсказку в `raw_hints`. Паттерны — регулярные выражения Python. После пуша в `main` правило начнёт работать на следующем PR.

### Стайлгайд

**Папка:** `style_guide/`

Пять markdown-файлов, которые описывают правила стиля для LLM-агента:

| Файл | Содержание |
|---|---|
| `01_instructions.md` | Как писать инструкции: глагол в начале, один шаг — одно действие |
| `02_neutral_tone.md` | Нейтральный тон: без оценок, без авторского «я» |
| `03_word_choice.md` | Выбор слов: конкретика вместо общих понятий |
| `04_sentences.md` | Структура предложений и абзацев |
| `05_links.md` | Оформление ссылок и перекрёстных ссылок |

LLM-агент получает весь стайлгайд при каждой проверке. Если вы хотите изменить требования к стилю — редактируйте эти файлы. Изменения вступят в силу сразу.

---

### LLM-агент

**Файл:** `.github/scripts/review-agent.py`

Агент объединяет всё собранное и формирует финальный результат. Он работает так:

1. Получает список изменённых файлов и читает их полностью (не только diff).
2. Получает diff PR — чтобы знать, какие строки добавлены.
3. Получает результаты линтера.
4. Загружает стайлгайд.
5. Отправляет всё это в LLM с инструкцией: найти нарушения стайлгайда, которые не поймал линтер.
6. Получает от LLM список замечаний в JSON.
7. Публикует комментарии в PR через GitHub API.

**Inline-комментарии** агент оставляет только на строках, которые есть в diff PR, — GitHub не разрешает комментировать строки, которые не изменялись. Нарушения в неизменённых строках агент выносит в общую сводку.

**Fallback-логика:** если LLM не ответил или вернул ошибку — агент публикует в PR только результаты линтера без LLM-анализа. Воркфлоу не падает.

---

### MCP-сервер

**Папка:** `mcp_server/`  
**Файл конфигурации деплоя:** `render.yaml`

MCP-сервер (Model Context Protocol) — опциональный компонент. Это небольшой веб-сервис на FastAPI, который хранит стайлгайд и правила линтера и раздаёт их по API.

**Зачем нужен:** если вы поддерживаете документацию в нескольких репозиториях, MCP-сервер становится единственным источником правил. Обновили стайлгайд — изменения автоматически применяются во всех репозиториях.

**Если MCP не настроен:** линтер и агент читают правила и стайлгайд напрямую из файлов в репозитории. Функциональность не меняется.

Сервер разворачивается на [Render.com](https://render.com) бесплатно. Инструкции — в [SETUP.md](./SETUP.md).

## Пошаговый сценарий

Разберём, что происходит от лейбла до комментариев.

**1. Триггер.** Автор вешает лейбл `docs-review` на PR. GitHub регистрирует событие `pull_request.labeled` и запускает воркфлоу.

**2. Подготовка.** Воркфлоу клонирует репозиторий, устанавливает Python-зависимости и скачивает Vale.

**3. Определение файлов.** Скрипт сравнивает HEAD ветки PR с базовой веткой и выбирает только `.mdx`-файлы из `docs/ru/` и `docs/en/`. Если таких файлов нет — воркфлоу завершается без ошибки.

**4. Линтинг.** Параллельно:
- Python-линтер проверяет русские файлы и записывает результат в JSON.
- Vale проверяет английские файлы и записывает результат в JSON.
- Оба JSON объединяются в один файл.

**5. Проверка MCP.** Если задан `MCP_SERVER_URL`, воркфлоу проверяет доступность сервера (три попытки с паузой 10 секунд). Результат передаётся агенту.

**6. LLM-анализ.** Агент читает файлы, загружает стайлгайд (с MCP или локально) и вызывает LLM. При недоступности LLM — три попытки с паузой. Если все неудачны — публикуется только сводка линтера.

**7. Публикация.** Агент отправляет комментарии в PR через GitHub API. Inline-комментарии — только на изменённых строках. Итоговая сводка — в теле ревью. Результаты линтера добавляются в сводку в виде раскрывающегося блока.

---

## Что видит автор в PR

- **Inline-комментарии** на конкретных строках — от линтера (точное совпадение) и от LLM (анализ контекста).
- **Сводка ревью** — общий вывод LLM о проверенных файлах.
- **Детали линтера** — список всех срабатываний в раскрывающемся блоке внутри сводки.

Все комментарии публикуются с типом `COMMENT`, а не `REQUEST_CHANGES` — воркфлоу не блокирует мёрж PR.

## Как изменить правила или стайлгайд

| Что изменить | Где |
|---|---|
| Добавить запрещённое слово | `styles/Russian/WordChoice.yml` |
| Добавить замену директивной конструкции | `styles/Russian/Substitutions.yml` |
| Изменить требования к стилю для LLM | `style_guide/*.md` |
| Изменить используемую модель | GitHub Variables → `LLM_MODEL` |
| Изменить максимальный размер файла для LLM | `review-agent.py` → константа `MAX_FILE_CHARS` |

После изменения файлов в `main` правила начнут действовать на следующем PR. Если настроен MCP-сервер, Render автоматически пересобирает сервис при пуше — задержка 1–2 минуты.
